本仓库采用「业务层抛出标准 .NET 异常 + 框架/工具层捕获并上报 Sentry」的分层错误处理模式，未定义自定义异常类型或统一错误码。

## 1. 使用的系统与工具
- Sentry SDK：通过 Services/SentryTelemetryService.cs 统一管理生命周期、面包屑（Breadcrumb）和事务（Transaction），提供 CaptureException、AddBreadcrumb、WithInstrumentation / WithInstrumentationAsync 等封装方法。
- Microsoft.Extensions.Logging：在插件入口和服务层使用 ILogger<T> 记录错误日志。
- ModelContextProtocol (MCP) 工具层：每个 IMcpServerTool 实现自行 try/catch 异常，将错误信息包装为结构化结果返回给调用方。

## 2. 关键文件与位置
- AgentIsland/Services/SentryTelemetryService.cs — 遥测服务，集中管理 Sentry 初始化/关闭、异常上报、事务包裹。
- AgentIsland/Plugin.cs — 插件生命周期入口，在 AppStarted/AppStopping 中捕获 MCP 服务器启停异常并上报。
- AgentIsland/Mcp/McpServerManager.cs — MCP 服务器启动/停止的 try/catch 边界，统一记录 Transaction 状态。
- AgentIsland/Automation/RunAcpAction.cs — 自动化动作入口，对配置不合法场景抛出 InvalidOperationException。
- AgentIsland/Mcp/Tools/*.cs（如 EchoCaveTool.cs、GetScheduleByDateTool.cs、SendNotificationTool.cs 等）— 各工具方法内部 try/catch，捕获后上报 Sentry 并以结构化结果返回错误消息。

## 3. 架构与约定
- 分层职责
  - 业务/自动化层（Automation/*、Services/*）：遇到参数非法、功能未启用、资源缺失等可预期错误时，直接 throw new InvalidOperationException / ArgumentException，由上层捕获。
  - 框架/基础设施层（Plugin.cs、McpServerManager.cs）：在进程级边界（启动/停止）捕获所有 Exception，记录日志 + 上报 Sentry，然后重新抛出，避免吞掉异常。
  - MCP 工具层（Mcp/Tools/*）：每个 CallTool 方法内 try/catch，捕获后将异常信息包装进结构化返回值，保证 MCP 协议不会因未处理异常崩溃；同时调用 telemetry.CaptureException 上报。
- 遥测集成约定
  - 所有关键路径使用 SentryTelemetryService.WithInstrumentation* 包裹，自动创建 Transaction、添加 Breadcrumb，并在 catch 分支标记 SpanStatus.InternalError。
  - 未启用遥测时（_sentryHandle is null），所有 API 均安全降级为空操作。
- 无全局中间件：没有统一的异常处理中间件或过滤器，错误处理分散在各调用点。

## 4. 开发者应遵循的规则
1. 业务校验抛标准异常：参数不合法抛 ArgumentException，运行时条件不满足抛 InvalidOperationException，不要返回空值或特殊字符串作为错误信号。
2. 在框架边界捕获并上报：插件生命周期、MCP 服务器启停等外部交互点必须 try/catch Exception，调用 _logger.LogError + _telemetry.CaptureException，必要时设置 Transaction 状态。
3. MCP 工具方法必须 try/catch：每个 CallTool 实现需捕获异常，上报 Sentry 后以结构化结果返回错误消息，确保协议稳定。
4. 优先使用 WithInstrumentation* 包裹异步/同步操作：减少重复样板代码，自动完成事务与面包屑记录。
5. 不要在工具层吞掉异常而不记录：即使返回结构化错误，也必须调用 CaptureException 以便线上排查。
6. 避免自定义异常类型：当前代码库未定义专用异常类，统一使用 .NET 内置异常即可。
